愿望清单与集卡盲盒激励系统 PRD

本文档定义 AiLifeOS 的跨模块激励系统。它借鉴低摩擦打卡、连贯反馈和可视化进度的产品思路，但不做付费抽奖和社交排名。系统通过有效打卡发放卡片，用户集卡后开启盲盒，盲盒奖励来自用户自己的心愿清单。

所属项目 AiLifeOS
模块类型 Core 跨模块能力
文档状态 产品设计基线
关键边界 非付费抽奖，非社交排名

1. 定位

activecore

愿望激励系统是 AiLifeOS 的 Core 能力，不属于训练模块或学习模块私有功能。

训练、学习、健康、工作等模块负责产出有效打卡、目标进度和里程碑事件；愿望激励系统统一负责发卡、集卡、盲盒、心愿清单、兑换和风控。

一句话定位 把长期坚持转化为可见卡片，再把卡片连接到用户真正想要的心愿奖励。

2. 设计原则

activeresponsible
  • 激励服务目标，不反过来绑架目标。
  • 卡片只能通过真实行动获得，不能付费购买。
  • 盲盒只抽用户自己的心愿清单，不做外部商业抽奖。
  • 大额心愿需要通过财务系统做预算确认和冷静期，不能自动触发消费。
  • 预算不足时优先推迟、转为心愿基金进度，谨慎允许预支。
  • 恢复、降级、合理跳过也可以算有效行为，避免为了集卡而过度训练。
  • 规则透明、可解释、可关闭，不制造连续签到焦虑。
  • 奖励要提升长期执行，不鼓励短期刷任务。

3. 核心闭环

activeloop
设定目标
  -> 创建心愿清单
  -> 完成有效打卡
  -> 获得卡片
  -> 收集到卡册
  -> 达到盲盒开启条件
  -> 从心愿清单抽取奖励
  -> 财务系统检查预算
  -> 用户确认预算、预支或推迟
  -> 兑换、推迟、预支或转为心愿进度
  -> 记录奖励历史，进入复盘

这个闭环连接的是“我今天做了什么”和“我真正想要什么”。它不是为了让用户沉迷抽卡，而是为长期目标提供情绪反馈和延迟满足机制。

4. 核心概念

activemodel
概念说明
有效打卡来自训练、学习、健康等模块的有效行为记录。跳过但说明合理原因，也可以是有效打卡。
卡片用户完成有效行为后获得的激励资产，可以按模块、主题、稀有度和周期归档。
卡册卡片集合，例如“训练适应期卡册”“学习冲刺卡册”“健康恢复卡册”。
心愿清单用户自己维护的奖励池，可以包含物品、旅行、体验、学习资源、服务等。
盲盒消耗卡片或达成阶段目标后开启，从合格心愿池中抽取奖励。
心愿进度大额心愿不一定一次解锁，可以通过多次盲盒增加进度。
心愿基金为某个心愿预留的预算进度，可以由用户手动存入或由盲盒结果转入。
预支预算不足但用户仍想兑换时，占用未来可自由支配预算的受控机制。
兑换用户确认预算、时机和必要条件后，将抽中的心愿转为实际行动。

5. 发卡机制

activecards

卡片类型

卡片获得条件设计目的
行动卡完成一次有效打卡。给每日行动即时反馈。
坚持卡连续完成计划内行动，或一周完成指定次数。强化稳定性，而不是强迫每天都做同一件事。
里程碑卡完成阶段目标，例如训练适应期第 1 周。强化阶段成果。
恢复卡按计划完成恢复、拉伸、休息或疼痛保护。避免用户为了集卡过度训练。
调整卡合理跳过、降级、改期并记录原因。把现实变化纳入系统，而不是惩罚变化。
突破卡达成个人最佳、完成困难任务、连续复盘。记录具有纪念意义的进步。

发卡规则

  • 每天可获得卡片数量有上限，防止刷任务。
  • 同一低价值行为不能反复刷卡。
  • 安全行为可以发卡，例如按疼痛规则停止训练。
  • 不同模块的卡片可以进入同一个总卡册，也可以保留模块主题卡册。
  • 卡片稀有度由任务难度、连续性、阶段意义和安全质量共同决定。

稀有度建议

稀有度来源用途
基础普通有效打卡。用于开启小盲盒。
进阶完成计划内训练、学习任务或完整复盘。用于标准盲盒或心愿进度。
稀有周目标完成、阶段目标完成、恢复策略执行良好。用于高价值盲盒。
纪念长期里程碑、个人突破、特殊节点。主要用于卡册收藏，不轻易消耗。

6. 心愿清单

activewishlist

心愿清单是盲盒奖励池。用户可以添加任何对自己有意义的奖励，不限制品类，但系统要帮助用户标记预算、优先级和适合的解锁条件。

字段说明
名称例如买一部手机、去旅行、买网球课、看一场演出。
类型物品、体验、旅行、学习、服务、休息、社交、自定义。
预估成本用于预算保护，不作为系统自动消费依据。
优先级用户主观想要程度。
等级小心愿、标准心愿、大心愿、长期心愿。
预算状态可立即兑换、需要确认、需要储蓄、暂不可兑换。
冷静期大额或冲动型心愿需要等待再确认。
适用模块可限定由训练、学习或综合卡片触发。
状态草稿、可进入盲盒、已抽中、已兑换、推迟、归档。

7. 盲盒机制

activecontrolled random

盲盒是一种可控随机反馈，不是赌博机制。它不接受付费购买次数，不提供现金奖励，不对外部奖品做商业抽奖。

盲盒等级开启条件示例奖池
小盲盒7 张基础卡，或一周完成 3 次有效打卡。低成本小心愿、休息体验、小奖励。
标准盲盒21 张卡，或完成一个周目标。标准心愿、学习资源、运动装备小件。
阶段盲盒完成阶段目标，例如训练适应期 4 周。高优先级心愿、旅行进度、大额心愿进度。
纪念盲盒半年目标完成、重大突破。长期心愿、重要体验、纪念级奖励。

抽取规则

  • 只有状态为“可进入盲盒”的心愿可以进入奖池。
  • 盲盒会按心愿等级、预算状态、优先级和冷静期筛选奖池。
  • 如果没有合格心愿，盲盒转为“心愿进度”或提示用户补充心愿清单。
  • 大额心愿优先抽取“进度”，不默认一次性兑换。
  • 抽取概率或权重必须可解释，至少对用户展示规则摘要。
  • 同一心愿抽中后进入冷却，避免短期重复出现。

8. 兑换机制

activebudget guard

抽中不等于自动消费。AiLifeOS 只能给出“你已经达成兑换资格”，实际购买、旅行或消费需要用户再次确认，并通过财务系统进行预算检查。

抽中心愿
  -> 展示达成条件和原因
  -> 财务系统检查预算状态、心愿基金、冷静期和预支资格
  -> 用户选择：立即兑换 / 推迟 / 转为进度 / 申请预支 / 放弃并返还部分卡片
  -> 记录兑换历史
  -> 周复盘记录奖励对执行的影响
  • 小心愿可以直接兑换，但仍需要用户确认。
  • 大心愿需要预算确认，可拆成多个心愿基金进度节点。
  • 预算不足时默认不建议兑换，优先转为心愿基金进度或推迟。
  • 用户选择预支时，必须显示未来预算影响、偿还计划和冷却期。
  • 用户可以把抽中的奖励推迟，系统不应该强迫立即消费。
  • 兑换记录应该进入复盘，判断激励是否真的提升执行。

9. 预算与预支

activebudget guard

愿望激励系统不能独立判断用户是否应该消费。所有心愿兑换都必须读取轻量财务系统的预算状态。

预算结果系统建议用户操作
预算充足允许兑换，但仍需确认。立即兑换 / 推迟。
预算接近上限提示谨慎，建议推迟或部分兑换。推迟 / 转心愿基金 / 继续确认。
预算不足不建议兑换。转心愿基金 / 推迟 / 申请预支。
安全储备不足禁止大额兑换和预支。只能推迟或转进度。
已有未还预支限制新的预支。先完成偿还计划。

预支规则

  • 预支不是额外收入，而是占用未来可自由支配预算。
  • 预支必须生成偿还计划，例如未来 4 周每周减少娱乐预算。
  • 预支有上限，不能超过未来可自由支配预算的一定比例。
  • 存在必要支出风险、安全储备不足或已有未还预支时，不允许新预支。
  • 预支成功后进入冷却期，冷却期内不能再开高价值盲盒。
  • 用户可以取消预支申请，心愿转回心愿基金进度。

详细预算规则见 docs/requirements/finance-budget-system-prd.html。

10. 跨模块接入

activeintegration

各模块不直接处理奖励逻辑，只发送标准事件给激励系统。

模块可产出事件可获得卡片
训练状态打卡、训练完成、恢复完成、疼痛保护、周目标完成。行动卡、恢复卡、里程碑卡、突破卡。
学习学习打卡、阅读完成、复习完成、阶段测验、项目交付。行动卡、坚持卡、里程碑卡。
健康睡眠记录、饮水、拉伸、恢复、疼痛记录。恢复卡、行动卡。
工作任务完成、项目推进、复盘完成。行动卡、突破卡。
日程按计划完成、合理改期、冲突处理。调整卡、坚持卡。

13. 风控规则

activeguardrails
  • 不允许付费购买卡片、抽奖次数或稀有度。
  • 不提供现金、返现、证券、虚拟博彩类奖励。
  • 大额心愿必须经过预算确认，必要时进入冷静期。
  • 预算不足时不默认兑换，必须推迟、转心愿基金或走预支流程。
  • 预支必须有上限、偿还计划和冷却期。
  • 安全储备不足时禁止预支和大额兑换。
  • 每日发卡有上限，防止刷低价值任务。
  • 训练模块必须把恢复、降级、疼痛停止视为有效安全行为。
  • 如果激励导致用户过度训练或焦虑，应允许关闭或降低激励强度。
  • 盲盒规则必须可解释，不能隐藏关键条件。
  • 奖励系统不能覆盖模块安全规则，例如不能为了卡片绕过训练安全规则。

14. MVP 范围

activeMVP
MVP 包含
  • 一个心愿清单。
  • 基础卡、恢复卡、里程碑卡三类卡片。
  • 训练模块打卡事件接入。
  • 小盲盒和标准盲盒。
  • 抽中心愿后的确认、推迟、转进度。
  • 基础预算保护、心愿基金、预支申请和每日发卡上限。
MVP 不包含
  • 多人排行榜。
  • 付费抽卡。
  • 复杂卡牌交易。
  • 外部商家奖品。
  • 自动购买或支付。
  • 完整记账系统。
  • 真实借贷、信用卡接口或金融账户同步。
  • 复杂概率运营系统。

15. 验收标准

activeacceptance
  1. 用户可以创建心愿清单，并为心愿设置类型、成本、优先级、等级和预算状态。
  2. 训练模块完成有效打卡后，激励系统能生成对应 RewardEvent 并发放卡片。
  3. 恢复、降级、合理跳过能获得安全导向的卡片或调整卡。
  4. 用户能查看卡册和盲盒进度。
  5. 达到条件后，用户能开启盲盒并从合格心愿池中获得结果。
  6. 大额心愿不会自动兑换，必须经过预算确认。
  7. 预算不足时，系统能提供推迟、转心愿基金或申请预支选项。
  8. 申请预支时，系统能展示未来预算影响和偿还计划。
  9. 安全储备不足或已有未还预支时，系统能阻止新的高风险预支。
  10. 用户可以选择兑换、推迟、转为进度或放弃。
  11. 每日发卡上限生效，不能无限刷卡。
  12. 系统明确展示盲盒规则摘要和安全边界。
  13. 用户可以关闭或降低激励系统强度。

16. 未决问题

draftquestions
  • 卡片是否需要视觉插图，是否用 image2 生成卡面。
  • 第一版盲盒是随机抽取，还是采用伪随机 + 保底机制。
  • 预支上限应该按月自由支配预算的固定比例，还是用户自定义。
  • 卡片消耗后是否仍保留在收藏卡册中。
  • 不同模块产出的卡片是否可以通用兑换，还是需要模块限定。
  • 是否需要“心愿基金”概念来累积大额奖励进度。
  • 财务系统第一版是否只做手动预算录入，不接入任何银行或支付平台。

维护规则：涉及愿望清单、集卡、盲盒、奖励兑换、卡片发放或跨模块激励时，必须同步更新本文档。
