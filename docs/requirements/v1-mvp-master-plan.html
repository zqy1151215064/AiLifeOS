1. 文档定位

baselinehandoff

本文档是 AiLifeOS V1 MVP 的总控交付基线，作用是把连续讨论中的产品想法落成可执行范围。

  • 产品层：明确 V1 做什么、不做什么、为什么做。
  • 需求层：把场景转成用户故事、功能点和验收标准。
  • 技术层：确定本地优先架构、模块边界、数据对象和服务接口。
  • 工程层：把功能拆成程序员可执行里程碑和任务。
执行规则 后续开发不能直接从聊天记忆开写代码，必须先对照本文档确认当前任务属于哪个模块、哪个优先级、哪个里程碑。

3. V1 定位

V1MVP

AiLifeOS V1 的目标不是做完整生活管理平台，而是先落地一个可闭环的个人生活操作系统最小版本。

围绕训练进步、饮食配合、灵活执行、持续打卡、愿望激励、预算约束，形成第一个可用闭环。

V1 重点解决以下问题：

  • 用户是健身小白，不知道今天该练什么、怎么练。
  • 用户有羽毛球、网球活动，训练计划不能和真实日程冲突。
  • 训练计划和饮食安排必须同步生成，而不是两个孤立模块。
  • 动作教学需要图文并茂，动作图通过 image2 按需生成并复用。
  • 打卡不是压力签到，而是让系统理解完成、跳过、调整、疼痛和疲劳。
  • 愿望激励把长期坚持和真实心愿连接起来。
  • Core 财务模块判断奖励是否能兑现，并支持手动记录开销。

4. 范围边界

in scopeout of scope
V1 包含
  • 个人档案：训练目标、运动基础、身体状态、饮食偏好、预算基础信息。
  • 日程输入：固定球局、临时网球课/羽毛球、今日可训练时间。
  • 训练计划：基于周任务池生成今日训练，不写死课表。
  • 饮食安排：根据今日训练和球类活动生成饮食建议。
  • 灵活调整：跳过、改期、缩短、替换训练，并记录原因。
  • 打卡记录：状态、训练、饮食、球类活动、计划变化、疼痛。
  • image2 动作图链路：生成 prompt、调用中转站、保存静态图片、复用。
  • 愿望清单、卡片、卡册、小盲盒和标准盲盒。
  • Core 财务：预算设置、预算检查、心愿基金、预支判断、手动开销记录。
V1 不包含
  • 用户注册、登录、多用户体系、云同步。
  • 真实支付、自动购买、自动借贷。
  • 支付宝、微信、淘宝、美团、银行等外部账单自动同步。
  • 完整记账 App、投资理财建议、复杂财务报表。
  • 社交、排行榜、好友监督。
  • 医疗诊断、康复治疗方案。
  • 大型食物数据库、扫码识别、拍照估算热量。
  • 预生成完整动作图库、移动端原生 App。

5. 用户故事

requirements
场景用户故事验收标准
健身小白不知道怎么练作为健身小白，我希望系统告诉我今天练什么、为什么练、怎么做，避免乱练和受伤。今日计划包含训练目标、动作列表、组数、次数、强度、注意事项和替代动作。
用户有羽毛球/网球安排作为经常打球的人，我希望系统把球类活动当作训练负荷，而不是普通备注。当天有网球课或羽毛球时，系统会调整力量训练强度、训练部位和恢复建议。
计划经常变化作为普通上班族，我希望今天不想练、没时间、临时有课时，可以调整计划。用户可以跳过、缩短、替换、改期训练，并生成新的今日训练和饮食建议。
训练需要配套饮食作为想提升运动表现的人，我希望训练计划同步告诉我今天怎么吃。每日计划包含早餐、午餐、训练前、训练后、晚餐和饮水建议。
动作看不懂作为新手，我希望动作有图文说明，减少错误动作。动作详情页有文字步骤、注意事项、常见错误、image2 教学图；已有图片可复用。
想持续坚持作为用户，我希望打卡不是压力签到，而是帮助系统理解我的真实执行。支持完成、部分完成、跳过、疼痛、疲劳、饮食执行等多种打卡结果。
需要激励作为用户，我希望坚持训练或学习后，可以获得和真实心愿相关的奖励。有效打卡能获得卡片，集卡达到规则后可开启盲盒，奖项来自心愿清单。
奖励不能乱花钱作为用户，我希望系统在兑换奖励前判断预算是否允许。兑换心愿前必须经过 Core 财务预算检查，输出可兑换、建议推迟、转基金、可预支或禁止兑换。
需要记录开销作为用户，我希望先手动记录支出，让预算判断有依据。用户可以新增、编辑、删除开销，记录金额、分类、日期、来源、支付方式、关联目标或心愿。

6. 功能模块

P0P1
模块优先级V1 功能点验收标准
个人档案P0身高体重、训练经验、目标、伤痛、饮食偏好、预算基础信息。系统可基于档案生成训练、饮食和预算判断。
日程与灵活计划P0固定球局、临时课程、今日可用时间、计划调整。今日计划能根据日程变化重新生成。
训练计划P0周任务池、今日训练、动作列表、强度控制、替代动作、训练后拉伸。用户每天能获得一个包含训练主体和训练后拉伸的可执行方案。
饮食安排P0按训练日、球类日、恢复日生成饮食建议。今日饮食会随训练变化同步调整。
打卡系统P0状态、训练、拉伸、饮食、球类、跳过、疼痛打卡。打卡结果能影响后续计划和激励事件。
image2 动作图P0动作 prompt 生成、图片生成、静态存储、复用。同一动作再次查看时优先读取已有图片。
愿望清单P0添加心愿、成本、优先级、类型、预算状态。盲盒奖池只能来自用户自己的心愿。
卡片与盲盒P1有效打卡发卡、卡册进度、盲盒开启。达到规则后可抽取合格心愿，不自动消费。
Core 财务P0预算设置、奖励预算、预算检查、心愿基金、预支判断。任何奖励兑换前必须输出财务判断。
手动开销P0手动新增、编辑、删除、分类、关联心愿或目标。开销会影响当期预算和兑换判断。
统计复盘P1训练完成率、饮食执行、打卡趋势、预算使用。用户能看到阶段性执行情况和调整建议。

7. 核心流程

loop

每日计划流程

读取个人档案
  -> 读取固定日程和临时日程
  -> 读取身体状态和历史打卡
  -> 从周任务池选择今日训练
  -> 生成今日饮食安排
  -> 展示今日计划
  -> 用户执行或调整
  -> 打卡记录结果
  -> 更新后续计划和激励进度

image2 动作图流程

用户打开动作详情
  -> 检查本地/静态资源是否已有动作图
  -> 有图：直接展示
  -> 无图：根据动作结构生成 image2 prompt
  -> 调用中转站 image2
  -> 保存图片、prompt、动作版本
  -> 后续同动作复用

愿望激励与财务流程

有效打卡
  -> 发放卡片
  -> 卡册进度增加
  -> 达到条件开启盲盒
  -> 从心愿清单抽取奖励
  -> Core 财务检查预算
  -> 预算够：允许用户确认兑换
  -> 预算不足：推迟 / 转心愿基金 / 申请预支
  -> 用户手动记录真实开销

8. 技术方案

local-firstReact

V1 推荐本地优先 Web App，先满足个人长期使用，不引入用户系统、云数据库和外部支付账单接入。项目采用 monorepo、单仓库、双进程架构：前端和 Go 后端都放在同一个 GitHub 仓库内，按应用和包分区管理，本地运行时分别启动。

React/Vite 前端
  -> 页面、交互、IndexedDB/Dexie 本地业务数据
  -> 通过 /api 和 /assets 访问 Go 服务

Go 本地服务
  -> image2 代理
  -> 图片落盘
  -> 静态图片访问
  -> 后续账单导入和本地 API
层面推荐方案理由
前端框架React + TypeScript + Vite适合快速构建本地个人应用，工程复杂度低。
路由React Router页面多但不需要 SSR。
状态管理Zustand比 Redux 简单，适合 MVP。
本地数据库IndexedDB + Dexie比 localStorage 更适合长期记录、资产 manifest、财务流水。
表单校验react-hook-form + zod减少表单样板代码，并保证数据结构可靠。
图表Recharts满足基础统计复盘，不做复杂 BI。
图标lucide-react轻量、统一、适合工具型界面。
后端服务同仓库内集成 Go 本地后端服务用于 image2 代理、图片落盘、静态图片访问、后续账单导入和本地 API，不默认使用 Node/Express。
image2Go 本地服务代理调用中转站API key 不能进前端，也不能提交到 GitHub。

9. 模块边界

coremodules
core/finance

管理收入、固定支出、自由预算、奖励预算、安全储备、开销记录、心愿基金、预支计划和预算检查。其他模块不能绕过它判断能不能花钱。

core/rewards

管理心愿清单、卡片、卡册、盲盒和兑换记录。抽中心愿后必须调用 core/finance 做预算检查。

core/logs

统一记录状态打卡、训练记录、饮食记录、疼痛记录、球类活动、计划变化和财务记录摘要。

modules/training

负责训练计划、饮食计划、动作库、安全规则、今日计划和打卡。它可以写日志、发 RewardEvent，但不能直接发卡或判断预算。

边界示例

训练完成打卡
  -> modules/training 计算完成度和安全状态
  -> core/logs 写入 CheckIn / TrainingLog
  -> core/rewards 接收 RewardEvent
  -> core/rewards 判断是否发卡
抽中心愿
  -> core/rewards 生成 DrawResult
  -> core/finance 执行预算检查
  -> 用户确认兑换 / 推迟 / 转心愿基金 / 申请预支
  -> core/logs 写入兑换结果

模块故障隔离

  • 模块之间只能通过明确 service contract 协作，不能跨模块直接读写内部状态。
  • Core 财务异常时，训练计划、打卡、动作详情、本地记录仍应可用；心愿兑换显示财务检查不可用并允许稍后重试。
  • image2 或 Go 服务不可用时，动作详情仍展示文字步骤、风险提示和 prompt，图片状态显示生成失败或待重试。
  • 愿望激励异常时，训练和打卡不受影响，RewardEvent 可以进入待处理队列。
  • 所有跨模块调用都要返回可解释错误，页面做局部降级，不做整页崩溃。

12. image2 安全链路

key boundaryasset
  • IMAGE_API_KEY 只放 .env.local。
  • .env.local 必须进入 .gitignore。
  • 前端不能直接请求 image2 或中转站。
  • 前端只能请求本地服务，例如 POST /api/image2/exercise-image。
  • 本地服务优先使用 Go 实现，读取 key，调用中转站，生成图片后保存静态资源。
  • 数据库只保存图片路径、prompt、状态、版本，不保存 key。
  • prompt 由结构化 Exercise 生成，不能让用户自由拼接敏感参数。
用户打开动作详情
  -> 前端查询 Asset
  -> missing/regenerate
  -> 前端请求本地服务
  -> Go 本地服务读取 .env.local
  -> 调用中转站 image2
  -> 保存图片
  -> 更新 Asset manifest / IndexedDB
  -> 前端展示静态图片

13. 财务与账单边界

manual firstfuture import

V1 财务模块是 Core 能力，但第一版只做手动预算和手动开销，不接外部账单自动同步。

MVP

手动添加开销，支持金额、分类、时间、支付方式、来源、备注、关联心愿或目标。

后续

支持 CSV、截图识别或账单文件导入，用于支付宝、微信、银行卡、信用卡等账单。

长期

条件允许时接入美团、淘宝、支付宝、银行等官方或可信账单来源，只读导入，不自动支付。

  • V1 所有开销来源默认为 manual。
  • 数据对象预留 source、sourceRecordId、importBatchId、reviewStatus。
  • 不保存明文账号、密码、支付密码或银行卡敏感信息。
  • 外部来源的数据只能辅助预算判断，用户可以手动修正最终分类和金额。

14. 工程里程碑

delivery
里程碑目标核心任务
0. 需求基线与工程约束锁定范围和技术方向。整理 V1 范围、非目标、模块依赖图、技术方案。
1. 数据模型与应用骨架建立可启动应用和核心模型。搭建 React/Vite、IndexedDB/Dexie、基础路由、核心类型。
2. 个人档案、日程、任务建立计划生成输入。个人档案、固定/临时日程、任务状态。
3. 训练 + 饮食闭环让用户每天知道练什么、怎么拉伸、怎么吃。周任务池、今日训练、训练后拉伸、饮食安排、计划调整。
4. image2 Prompt 与资产动作详情图文并茂。prompt 生成、本地服务调用、图片缓存、资产 manifest。
5. 打卡与记录系统低摩擦沉淀真实执行数据。状态/训练/饮食/球类/疼痛/计划变化打卡，统一记录流。
6. 愿望激励系统把有效行动转成卡片和心愿激励。心愿清单、卡片、卡册、小盲盒、标准盲盒。
7. Core 财务与手动开销预算守护奖励兑换。财务档案、预算、手动开销、心愿基金、FinanceDecision、预支。
8. 统计与复盘让用户看到进展和风险。训练统计、饮食执行、打卡趋势、预算使用、周复盘。

16. V1 完成标准

definition of done

V1 完成时，用户应该可以完整走完一次真实生活流程：

设置个人档案
  -> 设置每周三打球
  -> 生成今日训练、训练后拉伸和饮食
  -> 查看某个动作说明
  -> 生成动作教学图
  -> 完成或调整今日计划
  -> 打卡
  -> 获得卡片
  -> 积累卡片后开盲盒
  -> 抽中心愿
  -> 财务模块判断预算是否足够
  -> 用户选择兑现 / 推迟 / 转入心愿基金 / 预支
  -> 手动记录相关开销
  -> 在统计页看到训练、饮食、财务、激励进展
  • 用户不需要懂健身，也能知道今天该练什么、怎么练、练后怎么拉伸、怎么吃。
  • 用户临时有球局、没时间、不想练时，计划不会崩，而是能重排。
  • 用户能通过打卡留下真实执行数据，而不是只有成功/失败。
  • 动作图不预生成，按需生成并可复用，避免浪费 image2 调用。
  • 愿望激励能形成持续动力，但不会绕过预算。
  • Core 财务能作为跨模块基础能力，支撑奖励兑换和手动开销记录。
  • V1 在无用户系统、无云数据库、无支付、无外部账单接入的情况下仍能完整跑通。

维护规则：V1 范围、功能优先级、技术方案、模块边界或工程里程碑变化时，必须同步更新本文档。
