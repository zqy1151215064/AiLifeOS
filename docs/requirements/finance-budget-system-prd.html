Core 财务与预算模块 PRD

本文档定义 AiLifeOS 的 Core 财务模块。第一阶段不做完整记账 App，不接入银行和支付平台，重点服务手动开销记录、心愿兑换、预算判断、预支管理、心愿基金和跨模块财务状态。

所属项目 AiLifeOS
模块类型 Core 财务模块
文档状态 MVP 需求基线
关键边界 不自动消费，不自动借贷

1. 定位

activecore

财务系统是 AiLifeOS 的 Core 模块，模块路径建议为 core/finance。它不属于愿望激励系统，也不属于训练、学习、旅行等业务模块，而是所有涉及花钱、储蓄、预算、预支和消费复盘的模块共享的基础能力。

它的目标不是替代专业记账软件，而是在用户想兑换心愿、购买课程、购买装备、安排旅行或支付订阅时回答：钱花到哪里了，现在能不能买，需不需要攒，能不能预支，预支后未来怎么还。

Core 职责边界

  • 维护用户手动录入的财务状态。
  • 维护预算周期、奖励预算、安全储备和心愿基金。
  • 为其他模块提供预算检查结果，而不是让业务模块各自判断。
  • 记录与目标、心愿和模块关联的开销、消费或兑换结果。
  • 生成受控预支方案、偿还计划和冷却期。
  • 不直接决定训练计划、学习计划或奖励发放，只提供财务约束。

2. 设计原则

activeguardrails
  • 保护预算优先于奖励兑现。
  • 财务系统只做辅助决策，不提供专业投资、借贷或理财建议。
  • 不接入真实借贷，不自动消费，不自动扣款。
  • 预支不是额外收入，而是未来预算的提前占用。
  • 必要支出、安全储备和已有预支优先级高于心愿兑换。
  • 用户必须能看懂预算判断原因。

3. MVP 范围

activeMVP
MVP 包含
  • 手动录入月度收入、固定支出和可自由支配预算。
  • 安全储备状态。
  • 心愿基金。
  • 兑换前预算检查。
  • 预支申请、预支上限和偿还计划。
  • 手动开销、消费和兑换记录。
  • 面向其他模块的预算检查结果。
MVP 不包含
  • 银行账户同步。
  • 支付平台接入。
  • 信用卡账单自动识别。
  • 美团、淘宝、支付宝、银行等外部账单自动接入。
  • 投资组合管理。
  • 贷款申请或真实借贷。
  • 复杂会计分类。

4. 预算模型

activebudget
月度收入
  - 固定必要支出
  - 储蓄/安全储备
  - 已有预支偿还
  = 可自由支配预算

可自由支配预算
  - 已计划消费
  - 心愿基金存入
  = 当前可用奖励预算
预算项说明
月度收入用户手动录入的周期收入。
固定必要支出房租、通勤、基础餐饮、账单等不可轻易挪用支出。
安全储备用户希望保留的底线资金或紧急备用金。
可自由支配预算可以用于娱乐、体验、心愿和弹性消费的预算。
奖励预算可用于心愿兑换的预算，不等于全部自由支配预算。
预支占用已提前使用的未来奖励预算。

5. 心愿基金

activewishlist

心愿基金用于大额或长期心愿。盲盒抽中大额心愿时，可以不立即兑换，而是把本次奖励转化为心愿基金进度。

  • 每个心愿可以有独立基金目标。
  • 用户可以手动存入心愿基金。
  • 盲盒结果可以增加心愿基金进度。
  • 基金未满时默认不建议直接兑换大额心愿。
  • 用户可暂停、调整或归档心愿基金。

6. 预支机制

activecontrolled

预支允许用户在预算不足时提前使用未来的可自由支配预算，但必须受控。预支不代表系统鼓励消费，更不是借贷功能。

申请预支
  -> 检查安全储备
  -> 检查本月必要支出
  -> 检查已有预支
  -> 计算未来预算影响
  -> 生成偿还计划
  -> 用户确认
  -> 进入冷却期
规则说明
预支上限不能超过未来若干周期可自由支配预算的一定比例。
偿还计划预支后自动减少未来周期的奖励预算。
冷却期预支后一定时间内不能再次预支或开启高价值盲盒。
安全储备保护安全储备不足时禁止预支。
未还预支保护已有未还预支时限制新的预支。

7. 预算检查

activedecision
检查结果系统建议
可兑换预算充足，允许用户确认兑换。
谨慎兑换预算接近上限，建议推迟或部分转基金。
建议推迟预算不足，推荐转为心愿基金或下周期再兑换。
可申请预支预算不足但安全储备充足，可展示预支方案。
禁止预支安全储备不足、必要支出受影响或已有未还预支。

8. 开销记录与账单接入

activeexpense

开销记录是 Core 财务模块的一等能力。第一版先做用户手动添加，保证预算、心愿基金、预支和复盘有真实数据输入。

阶段能力边界
MVP手动添加开销，支持金额、分类、时间、支付方式、来源、备注、关联心愿或目标。不接外部账户，不自动同步。
后续支持 CSV、截图识别或账单文件导入，用于支付宝、微信、银行卡、信用卡等账单。导入后必须允许用户审核、修改和忽略。
长期在条件允许时接入美团、淘宝、支付宝、银行等官方或可信账单来源。只读导入，不保存明文敏感凭证，不自动支付。
  • 第一版开销必须支持手动新增、编辑、删除和分类。
  • 开销可以关联心愿、训练装备、学习课程、旅行计划或其他模块目标。
  • 外部账单接入必须是后续能力，不能阻塞 MVP。
  • 所有外部账单记录都要保留 source、sourceRecordId、importBatchId 和用户确认状态。
  • 外部来源的数据只能辅助预算判断，用户可以手动修正最终分类和金额。

11. 风控规则

activeguardrails
  • 财务系统不自动发起购买、支付或借贷。
  • 用户未设置预算时，不能给出“可放心兑换”的结论。
  • 安全储备不足时禁止预支和大额兑换建议。
  • 预支必须显示未来预算影响和偿还计划。
  • 已有未还预支时，默认不允许新的高价值预支。
  • 预算检查只做辅助，不替代专业财务建议。
  • 所有金额由用户手动维护，第一版不接入外部金融账户。
  • 后续接入外部账单时，只允许只读导入，不保存明文账号、密码、支付密码或银行卡敏感信息。
  • 导入记录必须可追溯来源，并允许用户删除导入批次。

12. 验收标准

activeacceptance
  1. 用户可以手动设置月收入、固定支出、安全储备和奖励预算比例。
  2. 系统能计算当前可自由支配预算和奖励预算。
  3. 用户可以手动添加、编辑、删除开销记录，并按分类和来源查看。
  4. 心愿清单中的心愿可以关联心愿基金。
  5. 兑换前系统能输出预算检查结果和原因。
  6. 预算不足时，系统能提供推迟、转心愿基金或申请预支选项。
  7. 申请预支时，系统能生成偿还计划和冷却期。
  8. 安全储备不足或已有未还预支时，系统能阻止高风险预支。
  9. 兑换或预支后能产生消费/预支记录。
  10. 第一版不依赖任何外部账单接口也能完整使用。
  11. 系统不触发任何真实支付、借贷或金融账户操作。

13. 未决问题

draftquestions
  • 预支上限默认比例是多少，是否允许用户自定义。
  • 心愿基金是否支持多个心愿并行。
  • 奖励预算是否应该独立于娱乐预算，还是作为娱乐预算的一部分。
  • 是否需要月度财务复盘页面。
  • 外部账单接入优先级：支付宝、微信、银行卡、信用卡、美团、淘宝，哪个先做。
  • 外部账单第一步采用 CSV/文件导入、截图识别，还是等待官方接口。
  • 未来是否接入真实记账或支付数据，当前默认不接入。
