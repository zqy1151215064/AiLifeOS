运动训练与饮食一体化 MVP PRD

本文档将 AiLifeOS 的首个真实场景转成可落地的产品需求。它面向 MVP，不覆盖完整 LifeOS，重点定义训练、饮食、记录、复盘、image2 动作图生成和安全规则。

1. 背景与定位

activeMVP

本模块是 AiLifeOS 的第一个落地场景，不是完整 LifeOS，也不是独立健身 App。

MVP 要服务一个明确场景：用户刚开通半年健身卡，是健身小白，平时打羽毛球和网球，希望通过健身房训练、饮食安排、动作教学和复盘，提升运动表现并建立训练习惯。

产品定位 面向健身小白、服务羽毛球和网球提升的训练与饮食一体化助手。

2. 用户问题

activeproblem
  • 不知道今天该不该练、练什么、怎么练。
  • 不认识健身房器械，不清楚动作标准和安全边界。
  • 训练计划容易被固定球局、临时网球课、疲劳、疼痛和临时事情打乱。
  • 有训练计划后，不知道当天应该怎么吃来配合训练。
  • 记录成本过高时容易中断，缺少低摩擦打卡机制。
  • 训练后缺少复盘，无法判断下次加量、保持、降级还是恢复。
  • AI 生成动作图可能不准确，需要可审核、可复用、可重生成。

3. MVP 目标

activegoal
  • 生成灵活的“今日训练 + 今日饮食”一体化计划。
  • 支持固定球局、临时网球课、跳过训练、缩短训练后的重新规划。
  • 每次训练后提供低门槛拉伸和放松安排，帮助恢复并降低不适风险。
  • 为每个动作提供小白可理解的图文教学。
  • 按需调用 image2 生成动作图，并保存为本地静态资源。
  • 设计低摩擦打卡机制，让状态、训练、饮食、疼痛和计划变化能持续沉淀。
  • 记录训练、饮食、疲劳、疼痛、球类活动。
  • 基于记录生成周复盘和下周调整建议。

4. 范围边界

activeYAGNI
MVP 包含
  • 个人档案、半年目标、固定日程、今日状态打卡。
  • 本周训练任务池，不写死固定课表。
  • 今日训练计划、训练后拉伸、今日饮食计划、动作详情、训练记录、饮食记录。
  • 今日状态、训练完成、拉伸完成、饮食执行、疼痛、球类活动和计划变化打卡。
  • image2 prompt 生成、图片资产状态管理。
  • 周复盘、训练安全规则、饮食联动规则。
MVP 不包含
  • 用户系统、云数据库、社区、排行榜、支付。
  • 多人训练计划、完整 LifeOS 其他模块。
  • 精确热量计算、食物数据库、拍照识别饮食。
  • 可穿戴设备接入。
  • 前端直接调用 image2。
  • 自动提交 GitHub。

5. 用户与场景

activepersona
场景用户意图系统响应
每天打开系统 快速知道今天练什么、怎么吃、有没有风险。 显示今日训练、饮食、推荐理由、状态风险和快捷调整。
早上或训练前打卡 用很低成本告诉系统今天身体状态和可用时间。 收集睡眠、疲劳、酸痛、疼痛、可训练时间和当日球类安排，用于生成或调整今日计划。
每周三固定打球 不要让健身计划和球局冲突。 固定事件优先，避开重腿或高冲突训练。
临时约了网球课 希望训练和饮食都随之调整。 降低肩部高负荷训练，增加训练前碳水和补水提醒。
今天不想练或没时间 不想计划崩掉。 提供跳过、改期、最低可行版本，并同步调整饮食。
不懂动作 需要看图和简单说明才能安全完成。 展示动作图、步骤、常见错误、停止信号和替代动作。

6. 核心流程

activeflow

主流程

首次建档
  -> 设置半年目标、固定球局、训练偏好、伤病限制
  -> 生成本周训练任务池
  -> 每日状态打卡
  -> 结合日程、疲劳、疼痛、球类活动生成今日训练和饮食
  -> 用户查看动作详情
  -> 无动作图时生成 prompt 并调用 image2
  -> 保存图片资产并展示
  -> 用户完成训练和饮食记录
  -> 用户完成训练后打卡和饮食执行打卡
  -> 系统复盘并调整后续计划

计划变化流程

用户触发变化
  -> 选择原因：不练 / 改天 / 只有 20 分钟 / 晚上打球 / 身体不舒服
  -> 系统重新评估训练任务池、日程、状态和风险
  -> 生成新的训练建议
  -> 同步生成新的饮食建议
  -> 将变化写入打卡记录，用于周复盘

打卡流程

打开今日页
  -> 状态打卡：睡眠 / 疲劳 / 酸痛 / 疼痛 / 可用时间 / 今日球类
  -> 系统生成或调整今日计划
  -> 执行训练和饮食
  -> 完成打卡：训练完成度 / RPE / 疼痛 / 饮食执行 / 备注
  -> 数据进入记录和周复盘

动作图生成流程

用户打开动作详情
  -> 查询 Asset manifest
  -> 有 ready 图片：直接展示
  -> 无图片或状态为 regenerate：生成 prompt
  -> 本地服务调用 image2
  -> 保存到 public/exercise-images/
  -> 更新 Asset 状态
  -> 用户可标记 rejected 或重新生成

10. 规则体系

activesafety

计划规则

  • 周计划是任务池，每日动态选择，不写死课表。
  • 固定/临时日程优先于训练计划。
  • 用户选择“不练”“改天”“只有 20 分钟”时，训练和饮食同步重排。
  • 每个训练提供完整版、标准版、最低可行版。

训练安全规则

  • 关节刺痛、麻木、尖锐疼痛时停止动作。
  • 当天高强度羽毛球，不安排重腿。
  • 当天网球课，不安排肩部高负荷训练。
  • 连续高疲劳时降级或恢复。
  • 动作变形时不加重量。
  • AI 只能解释规则，不能绕过规则。

饮食联动规则

  • 今日饮食由训练类型、强度、时间决定。
  • 训练日包含训练前/训练后补给建议。
  • 休息或跳过训练时取消训练补给，保持蛋白质和正常饮食。
  • 球类日强调碳水、水分、电解质。
  • MVP 不做疾病饮食和精确营养处方。

打卡规则

  • 打卡优先低摩擦，不要求用户输入长文本。
  • 状态打卡影响今日计划生成，完成打卡影响周复盘和后续任务池。
  • 连续漏打卡不惩罚用户，只在复盘中提示数据不足。
  • 跳过训练也算一种有效打卡，需要记录原因。
  • 疼痛打卡优先级高于完成率，出现关节痛时触发安全规则。
  • 打卡不做社交排名，不做连续天数焦虑设计。
  • 有效打卡可以向跨模块激励系统产出 RewardEvent，但训练安全规则优先于发卡规则。

11. image2 需求

activesecret safe
  • 前端不直接调用 image2。
  • key 只放 .env.local 或本地服务，不能提交。
  • 动作图由结构化动作数据生成 prompt。
  • 图片状态支持 missing、generating、ready、rejected、regenerate。
  • AI 图不是动作标准的唯一依据，用户可标记不准确并重生成。
  • 生成图片保存到 public/exercise-images/，后续同动作复用。

12. 打卡机制

activecheck-in

打卡机制是记录系统的低摩擦入口，不是为了制造连续天数压力。它的核心价值是让系统获得足够上下文，支持今日计划、计划重排、训练饮食联动和周复盘。

打卡类型触发时机核心字段用途
今日状态打卡 打开今日页、训练前 睡眠、疲劳、酸痛、疼痛、可用时间、今日是否打球 决定今日训练强度、是否降级、饮食是否调整。
训练完成打卡 训练后 完成度、RPE、重量/次数摘要、疼痛、备注 判断下次加量、保持、降级或替换动作。
饮食执行打卡 餐后、睡前或训练后 蛋白质、碳水、蔬菜、饮水、训练前后补给 判断恢复风险和饮食是否支持训练。
球类活动打卡 羽毛球/网球后 项目、时长、强度、表现感受、疼痛 影响后续训练负荷和恢复安排。
计划变化打卡 跳过、改期、缩短、替换时 变化类型、原因、是否改期、备注 让周复盘知道计划为什么变化，避免误判为单纯未完成。
疼痛打卡 出现不适时随时记录 部位、程度、疼痛类型、触发动作、是否停止 触发安全规则，调整动作和训练安排。

交互原则

  • 状态打卡控制在 30 秒内完成。
  • 训练完成打卡优先使用滑块、选择项、快捷按钮，备注可选。
  • 允许补打卡，但要保留实际发生日期。
  • 允许“跳过并说明原因”，跳过不是失败，而是计划重排输入。
  • 首页展示打卡状态，但不做强刺激连续打卡排名。

13. 验收标准

activeacceptance
  1. 用户完成建档后，系统能生成本周任务池和今日计划。
  2. 今日计划同时包含训练、训练后拉伸和饮食，并给出推荐理由。
  3. 用户能在今日页完成状态打卡，打卡结果能影响今日计划。
  4. 添加“今晚网球课”后，系统能调整训练和饮食。
  5. 点击“今天不练”后，任务回到任务池，饮食同步改为普通日。
  6. 跳过、改期、缩短训练时，系统能记录计划变化打卡和原因。
  7. 动作详情能展示步骤、风险、停止信号、替代动作。
  8. 没有动作图时能生成 prompt，并进入图片生成/缓存流程。
  9. 训练后能记录重量、次数、RPE、疼痛、拉伸完成情况和备注。
  10. 饮食能按餐次做轻量记录，不要求称重。
  11. 球类活动和疼痛能作为独立打卡进入记录系统。
  12. 周复盘能输出完成率、疲劳/疼痛风险和下周建议。
  13. 所有训练与饮食建议都显示安全提示：不替代医生、营养师或专业教练意见。

14. 成功指标

draftmetrics
  • 用户每周能完成至少一次训练记录和三次状态打卡。
  • 今日计划能在 1 分钟内让用户理解“练什么、怎么吃、为什么”。
  • 遇到计划变化时，用户能在不手动重排整周计划的情况下得到新建议。
  • 动作详情能让用户明确知道动作步骤、错误和停止信号。
  • 周复盘能输出至少 1 条可执行的下周调整建议。
  • 打卡流程不阻碍使用，状态打卡应能在 30 秒内完成。

15. 未决问题

draftquestions
  • 训练动作库第一批应该包含多少动作，是否先控制在 20-30 个。
  • image2 中转站的实际接口路径、返回格式和尺寸参数需要验证。
  • 第一版数据存储使用 localStorage、IndexedDB 还是项目内静态 JSON，需要开发前定稿。
  • 周复盘是规则生成、AI 生成，还是规则 + AI 混合生成。
  • 是否需要把 PRD 进一步拆成页面规格文档和数据模型文档。
  • 打卡是否需要单独页面，还是完全融入今日页、记录页和训练完成页。
  • 是否需要为打卡设计温和提醒，提醒频率和关闭机制如何设定。

16. 实施备注

activeengineering
  • 开发前必须先读 docs/product-memory.html。
  • 本 PRD 是训练与饮食模块的产品需求基线，开发时不能把模块做成独立健身 App。
  • 涉及训练、饮食、健康建议时，必须保留安全提示和保守策略。
  • 涉及 image2 时，必须遵守密钥不进前端、不进 GitHub 的规则。
  • 涉及愿望清单、集卡、盲盒和奖励兑换时，必须阅读 docs/requirements/reward-wishlist-system-prd.html。
  • 新增功能如果不能进入目标、计划、执行、记录、复盘、调整闭环，默认不做。

维护规则：当训练与饮食模块的范围、流程、数据对象、规则或验收标准变化时，必须同步更新本文档。
