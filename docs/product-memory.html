1. 产品定位

activecore

AiLifeOS 不是单一健身 App，而是一个面向个人长期目标的生活操作系统。

围绕个人目标，持续完成规划、执行、记录、复盘、调整和记忆沉淀的本地优先生活操作系统。

当前 V1 以运动训练与饮食一体化为首个业务场景，同时接入打卡、愿望激励、Core 财务和手动开销，跑通第一个完整闭环。

V1 服务的目标是：用户刚开通半年健身卡，是健身小白，平时打羽毛球和网球，希望通过训练计划、动作教学、饮食安排、打卡复盘、心愿激励和预算守护，让羽毛球、网球和整体生活执行能力持续提升。

2. 总体闭环

activesystem loop

AiLifeOS 的所有模块最终都应该遵循同一个闭环：

目标
  -> 计划
  -> 今日行动
  -> 执行记录
  -> AI 复盘
  -> 自动调整
  -> 个人记忆沉淀

运动训练模块当前的闭环是：

个人档案和半年目标
  -> 生成阶段计划和本周任务池
  -> 根据日程、身体状态、球类活动生成今日训练和饮食计划
  -> 用户查看动作图文教学
  -> 没有动作图时按需调用 image2 生成并保存为静态资源
  -> 用户完成训练、拉伸和饮食记录
  -> 系统根据疲劳、疼痛、完成情况和球类表现调整后续计划

3. 当前模块边界

activeboundary

当前 V1 名称：运动训练与饮食一体化 + 愿望激励 + Core 财务闭环。

它不是完整 AiLifeOS 的全部，只是第一个可用版本。后续可能加入学习、工作、知识库、习惯、项目管理等模块，但 V1 只实现训练饮食、打卡、激励和财务预算守护。

V1 总控交付文档见 docs/requirements/v1-mvp-master-plan.html。后续编码前应先读该文档，再读具体模块 PRD。

4. 用户画像

activepersonal use

当前优先服务的用户就是项目所有者本人：

  • 健身小白，不熟悉训练动作和健身房器械。
  • 刚开通半年健身卡，有明确的 24 周时间边界。
  • 平时打羽毛球和网球。
  • 希望提升移动、爆发、核心、肩肘稳定、下肢力量和整体运动表现。
  • 需要系统告诉自己今天练什么、怎么练、为什么这么练、怎么吃、练完如何调整。
  • 不需要社区、排行榜、复杂商业化功能。

5. 半年训练目标

active24 weeks

半年目标不是单纯增肌或减脂，而是：

通过安全、可持续的健身房训练和饮食安排，提升羽毛球和网球表现，同时建立长期训练习惯。
阶段周期目标
适应期第 1-4 周学动作、熟悉器械、建立习惯、避免受伤
基础期第 5-12 周提升基础力量、核心稳定和关节控制
专项期第 13-20 周将力量、移动、旋转能力转化到羽毛球和网球
巩固期第 21-24 周巩固表现、降低伤病风险、总结长期策略
MVP 决策 第一版优先认真做好前 4 周。后续阶段可以先保留方向，不需要一次性写死所有训练细节。

6. 灵活计划机制

activeplanning

计划不应是固定课表，而应是可重排的建议系统。

长期目标稳定，周计划是任务池，每日计划动态生成。

不推荐固定写死：

周一练腿，周三练胸，周五练背。

推荐生成本周任务池：

本周需要完成：
- 下肢基础 1 次
- 上肢稳定 1 次
- 核心抗旋转 1 次
- 灵活性恢复 2 次

每天根据以下因素选择最合适任务：

  • 今天是否有羽毛球、网球或网球课。
  • 昨天练了什么。
  • 睡眠和疲劳情况。
  • 是否有肌肉酸痛或关节疼痛。
  • 今天可训练时间。
  • 本周还剩哪些任务。
  • 距离本周结束还有几天。

7. 固定事件和临时变化

activeschedule first

用户可能有固定球局，也可能临时约网球课，或者当天突然不想练。这些变化必须是一等公民，而不是异常情况。

固定事件示例

每周三下班后和朋友打球。

临时事件示例

今天晚上约了网球课。

系统应该先处理固定日程和临时日程，再安排训练：

固定日程
  -> 临时日程
  -> 今日身体状态
  -> 本周任务池
  -> 今日训练和饮食建议

今日页需要支持这些操作：

  • 开始训练。
  • 换一个训练。
  • 改到明天。
  • 今天不练。
  • 我只有 20 分钟。
  • 我晚上要打球。
  • 身体不舒服。

每个操作都应该触发重新规划，而不是简单删除计划。

8. 跳过和降级机制

activeflexible

当用户点击“今天不练”时，系统应该轻量询问原因：太累、没时间、身体不舒服、临时有事、单纯不想练。

原因系统反应
太累安排恢复，降低本周负荷
没时间推荐 10-15 分钟最低可行版本
身体不舒服记录疼痛或不适，避开相关动作
临时有事将任务放回任务池并重新排期
单纯不想练允许跳过，记录趋势，不惩罚用户
版本时长用途
完整版45-60 分钟状态好、时间充足
标准版30-40 分钟日常默认
最低可行版10-15 分钟忙、累、不想练但希望保持连续性

9. 球类运动联动

activesafety

羽毛球和网球本身也是训练负荷，不能当成普通备注。

  • 当天有高强度羽毛球，不安排重腿。
  • 当天有网球课，不安排肩部高负荷训练。
  • 球类后第二天疲劳高，降低健身房强度。
  • 连续两天球类，第三天优先恢复。
  • 肩部疼痛时减少推举、扣杀相关强化和高负荷上肢训练。
  • 膝盖疼痛时减少跳跃、重深蹲、剧烈变向训练。

AI 可以解释原因，但这些安全规则应由明确规则控制，不完全交给自由生成。

10. 训练和饮食一体化

activedaily plan

每日计划必须同时包含训练和饮食安排。不是先生成训练，再给泛泛饮食建议，而是：

今日训练类型、强度和时间
  -> 决定今日饮食重点
  -> 生成早餐、午餐、训练前、训练后、晚餐和饮水建议

今日计划结构：

今日计划
  |- 训练安排
  |   |- 训练类型
  |   |- 动作、组数、次数、休息
  |   |- 新手重量建议
  |   `- 风险提示
  |- 训练后拉伸
  |   |- 肩颈/胸椎/髋/腿部等拉伸动作
  |   |- 每个动作时长
  |   |- 停止信号
  `- 饮食安排
      |- 早餐
      |- 午餐
      |- 训练前补给
      |- 训练后补给
      |- 晚餐
      |- 饮水建议
      `- 避免事项
  • 如果今天不练，取消训练前后补给，改成普通日饮食。
  • 如果临时增加网球课，增加训练前碳水和补水提醒。
  • 如果今天是下肢训练，保证碳水和蛋白质。
  • 训练后必须有 5-10 分钟拉伸或放松任务，按今日训练和球类活动选择部位。
  • 如果今天是恢复日，保持蛋白质，碳水适中，避免暴食。

第一版不做精确卡路里和食物数据库，优先使用小白友好的结构化餐次建议。

11. 饮食建议原则

activebeginner

当前饮食目标是支持运动表现和训练恢复，不是极端减脂。

运动表现优先，同时轻微改善体态。

记录方式可以使用手掌法，不要求称重：

  • 蛋白质：一掌心。
  • 碳水：一拳头。
  • 蔬菜：一到两拳头。
  • 脂肪：一拇指。
  • 饮水：按杯数或大致容量记录。

第一版饮食模块不做精确热量计算、大型食物数据库、条形码扫描、菜谱商城、复杂营养素报表、AI 拍照识别热量和严格减脂计划。

12. 动作教学需求

activeimage rich

由于用户是健身小白，动作图文教学是核心体验，不是附属说明。

动作详情页应包含
  • 标准动作图、起始姿势图、发力过程图、结束姿势图、常见错误图。
  • 训练目的、器械说明、文字步骤、新手重量选择。
  • 对羽毛球和网球的帮助、常见错误、停止信号、替代动作。
说明必须回答
  • 怎么站或怎么坐，手脚放哪里。
  • 身体哪里发力，做到什么程度。
  • 什么感觉是正常，什么疼痛必须停止。
  • 做不了时换哪个动作。

13. image2 动作图生成链路

activeon demand

当前不预先生成全部动作图。采用按需生成和静态资产缓存：

用户点击动作
  -> 检查本地是否已有该动作图片
  -> 有：直接展示静态图片
  -> 没有：根据结构化动作数据生成 image2 prompt
  -> Go 本地服务调用中转站 image2
  -> 保存图片到 public/exercise-images/
  -> 更新图片 manifest
  -> 页面展示图片
  -> 后续同动作直接复用

图片不要存数据库。现阶段个人使用，生成图片作为项目静态资源保存，并可提交到 GitHub 仓库。

14. image2 安全和密钥规则

activesecret safe

用户没有官方 image2，只持有中转站 key。中转站可能兼容 OpenAI 风格接口，也可能有差异，开发前需要先用最小脚本验证。

  • API key 不能写入前端代码。
  • API key 不能提交到 GitHub。
  • API key 放在 .env.local 或本地私有配置中。
  • 前端不能直接调用 image2。
  • 调用 image2 应通过本地 Go 后端服务完成；临时验证脚本也优先使用 Go。
  • 生成图片后保存到项目静态目录，App 运行时只读静态图片。

需要验证的中转站信息：base URL、模型名是否为 image2、图片生成接口路径、返回格式是 base64 还是 url、支持的尺寸和参数。

15. Prompt 生成原则

activeprompt

image2 只负责画图，不负责决定动作是否正确。动作定义必须来自结构化动作库。

动作名
适合等级
器械
目标肌群
动作步骤
常见错误
停止信号
替代动作
羽毛球收益
网球收益

Prompt 风格要求：

  • 干净白色背景。
  • 真实人体比例。
  • 运动康复教学风格。
  • 不要夸张肌肉。
  • 不要漫画风。
  • 不要复杂背景。
  • 不要错误器械。
  • 清晰表达起始姿势、过程或错误示范。

16. 打卡机制

activecheck-in

打卡机制是记录系统的低摩擦入口，不是为了制造连续天数压力。它用于让系统获得足够上下文，支持今日计划、计划重排、训练饮食联动和周复盘。

打卡类型作用
今日状态打卡记录睡眠、疲劳、酸痛、疼痛、可用时间和今日球类安排。
训练完成打卡记录完成度、RPE、疼痛、重量/次数摘要和备注。
饮食执行打卡按餐次轻量记录蛋白质、碳水、蔬菜、饮水和训练补给。
球类活动打卡记录羽毛球/网球时长、强度、表现感受和疼痛。
计划变化打卡记录跳过、改期、缩短、替换训练的原因。
疼痛打卡记录疼痛部位、程度、类型、触发动作和是否停止。
  • 状态打卡应能在 30 秒内完成。
  • 跳过训练也算有效打卡，需要记录原因。
  • 连续漏打卡不惩罚用户，只在复盘中提示数据不足。
  • 疼痛打卡优先级高于完成率，出现关节痛时触发安全规则。
  • 不做社交排名，不做强刺激连续打卡焦虑设计。

17. 愿望激励系统

draftcross-module

AiLifeOS 后续需要一套跨模块激励系统，把训练、学习、健康、工作等模块的有效打卡转化为卡片，再通过集卡和盲盒连接用户自己的心愿清单。

有效打卡
  -> 获得卡片
  -> 集齐卡片
  -> 开启盲盒
  -> 命中心愿清单奖励
  -> 用户确认预算和时机
  -> 兑换或推迟
  • 训练、学习等模块只产出有效打卡、目标进度和里程碑事件。
  • 激励系统统一负责发卡、集卡、盲盒、心愿清单和兑换规则。
  • 卡片只能通过真实行动获得，不能付费购买。
  • 盲盒奖项来自用户自己的心愿清单，不做外部商业抽奖。
  • 大额心愿需要接入 Core 财务模块做预算确认，不自动触发消费。
  • 预算不足时可以推迟、转为心愿基金进度，或在安全规则内申请预支。
  • 恢复、降级、合理跳过也可以算有效行为，避免为了集卡而过度训练。

详细需求见 docs/requirements/reward-wishlist-system-prd.html。

18. Core 财务模块

draftcore/finance

AiLifeOS 需要一套独立的 Core 财务模块，统一管理预算状态、心愿基金、预支、开销记录和兑换风险。它不是愿望激励系统的附属功能，而是所有涉及花钱决策的模块都要调用的基础能力。

第一阶段不做完整记账 App，不接银行、不接支付平台，优先服务手动开销记录、心愿兑换、预算判断、预支判断和消费复盘。

抽中心愿
  -> 财务系统检查预算
  -> 预算够：允许确认兑换
  -> 预算不够：推迟 / 转为心愿基金 / 申请预支
  -> 预支通过：生成偿还计划和冷却期
  -> 预支不通过：保留心愿进度
  • 记录月度可自由支配预算、必要支出、安全储备和心愿基金。
  • 第一版开销由用户手动添加，记录金额、分类、来源、时间和关联目标。
  • 大额心愿兑换前必须做预算检查。
  • 学习课程、训练装备、旅行计划、订阅工具等未来模块支出，也应通过同一套预算检查。
  • 长期可以接入美团、淘宝、支付宝、银行和信用卡账单，但必须作为只读导入，不自动支付、不自动借贷。
  • 预支不是额外收入，而是占用未来预算，必须生成偿还计划。
  • 预支需要上限、冷却期和风险提示。
  • 财务系统不替代专业财务建议，不自动消费、不自动借贷。

详细需求见 docs/requirements/finance-budget-system-prd.html。

19. 页面设计方向

activeui

AiLifeOS 的界面要像个人生活操作系统，不像普通后台，也不像花哨健身 App。

安静、克制、清爽、专业、有信息密度、有秩序感。
左侧导航栏
  |- 今日
  |- 目标
  |- 日程
  |- 任务
  |- 记录
  |- 统计
  |- 资源
  `- 设置

顶部状态栏
  |- 今天日期
  |- 当前目标阶段
  |- 今日状态
  `- 快捷操作

主内容区
  `- 当前页面内容
  • 不做营销落地页。
  • 不用大面积渐变和装饰性背景。
  • 不用巨型卡片堆砌。
  • 卡片圆角控制在 8px 以内。
  • 少阴影，多使用细边框、分区、留白和清晰层级。
  • 字体紧凑清晰，标题不要过大。
  • 图标优先使用统一线性图标风格。
  • 页面要适合长期高频使用。

20. 关键页面

activepages
页面作用
今日页打开后直接看到今日训练、训练后拉伸、饮食、状态打卡、风险和快捷操作
训练页展示本周训练任务池、今日训练和可替换训练
动作详情页展示图文教学、错误提示、停止信号和替代动作
饮食页展示与今日训练联动的餐次安排和饮食记录
日程页展示固定球局、临时课程、训练安排和冲突提醒
记录页查看训练、饮食、疲劳、疼痛、球类活动和打卡时间线
统计页展示完成率、训练负荷、疲劳趋势、疼痛次数、球类频率
资源页管理 image2 生成图片、prompt、状态和重生成
个人档案页配置身体信息、运动背景、伤病、目标和偏好
财务页查看预算、心愿基金、预支记录和兑换风险

今日页是最重要入口。今日页需要回答：今天该不该练、今天练什么、今天怎么吃、为什么这么安排、有没有风险、如果计划变了怎么调整。

21. 安全规则

activesafety

运动训练涉及身体风险，安全层必须明确。

  • 关节刺痛、麻木、尖锐疼痛时停止动作。
  • 连续高疲劳时降低训练量或安排恢复。
  • 当天有高强度球类活动时，不安排重腿。
  • 肩痛时避免高负荷推举、过量扣杀相关强化和大重量上肢训练。
  • 膝痛时避免跳跃、重深蹲和剧烈变向。
  • 动作变形时不要加重量。
  • 新手重量以“最后 2 次吃力但动作不变形”为准。

AI 可以提供解释和建议，但不能绕过安全规则。

23. 技术方向初稿

draftarchitecture

后续开发可以按以下思路组织，但不要求一次性全部实现。项目采用 monorepo、单仓库、双进程架构：V1 前端固定使用 React + TypeScript + Vite，Go 本地服务集成在同一个仓库中，本地运行时分别启动。

    src/
      core/
        profile/
        goals/
        schedule/
        tasks/
        logs/
        assets/
        reviews/
        rules/
        rewards/
        finance/
      modules/
        training/
      pages/
      shared/

  server-go/
    cmd/
      ailifeos/
    internal/
      image2/
      assets/
      config/
    storage/

packages/
  schemas/
  fixtures/

前后端分工：

React/Vite 前端
  -> 页面、交互、IndexedDB/Dexie 本地业务数据
  -> 通过 /api 和 /assets 访问 Go 服务

Go 本地服务
  -> image2 代理
  -> 图片落盘到 apps/server-go/storage/exercise-images/
  -> 暴露 /assets/exercise-images/ 静态访问
  -> 后续账单导入和本地 API

模块故障隔离原则：

  • 模块之间通过 service contract 协作，不能跨模块直接读写内部状态。
  • Core 财务异常时，训练、打卡、动作详情和本地记录仍应可用。
  • image2 或 Go 服务不可用时，动作详情仍展示文字教学和 prompt，图片生成进入待重试状态。
  • 愿望激励异常时，训练和打卡不受影响，RewardEvent 可以进入待处理队列。
  • 页面必须做局部降级，不因为单个模块错误导致整个 App 不可用。

静态资源建议：

public/
  exercise-images/

本地脚本建议：

scripts/
  test-image2.go
  generate-exercise-image.go

环境变量建议：

IMAGE_API_BASE_URL=
IMAGE_API_KEY=
IMAGE_MODEL=image2

.env.local 必须加入 .gitignore。

24. 开发优先级

activeMVP
  1. 产品框架和今日页。
  2. 个人档案和半年目标。
  3. 灵活本周任务池。
  4. 今日训练、训练后拉伸和饮食一体化计划。

  6. 动作详情页和结构化动作库。
  7. image2 prompt 生成和本地保存链路。

  11. V1 接入跨模块愿望清单、集卡和盲盒激励系统。
  12. V1 接入 Core 财务模块，为心愿兑换和跨模块付费决策提供预算检查、手动开销和预支管理。

25. 判断一个功能是否应该做

activeKISSYAGNI

新增功能前先问：

  • 它是否服务当前半年运动目标。
  • 它是否能进入目标、计划、执行、记录、复盘、调整闭环。
  • 它是否对健身小白有直接帮助。
  • 它是否能提升训练和饮食执行率。
  • 它是否会明显增加当前复杂度。

如果不能直接服务当前闭环，默认不做。

26. 当前一句话总结

activesummary
AiLifeOS V1 当前先做第一个完整闭环：以运动训练与饮食一体化为首个业务场景，根据半年运动目标、固定球局、临时安排、身体状态打卡和训练记录动态生成今日训练、训练后拉伸与饮食计划；动作图按需调用 image2 生成并保存为静态资源；训练、拉伸、饮食、球类、疼痛和计划变化通过低摩擦打卡进入记录与复盘；有效打卡进入愿望激励系统；心愿兑换必须经过 Core 财务预算检查，并支持手动开销记录、心愿基金和受控预支。

维护规则：修改产品方向、模块边界、训练饮食流程、image2 链路、页面设计或 MVP 范围时，必须同步更新本文档。
