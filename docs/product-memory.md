# AiLifeOS 产品记忆

本文档记录 AiLifeOS 当前阶段的产品共识。后续 AI 或开发者在继续设计、开发、重构前，必须先阅读本文档，避免忘记已有决策或把项目方向做偏。

## 1. 产品定位

AiLifeOS 不是单一健身 App，而是一个面向个人长期目标的生活操作系统。

核心定义：

```text
围绕个人目标，持续完成规划、执行、记录、复盘、调整和记忆沉淀的本地优先生活操作系统。
```

当前阶段只落地第一个真实模块：运动训练与饮食一体化模块。

该模块服务的目标是：用户刚开通半年健身卡，是健身小白，平时打羽毛球和网球，希望通过训练计划、动作教学、饮食安排和复盘，让羽毛球、网球和整体运动能力持续提升。

## 2. 总体闭环

AiLifeOS 的所有模块最终都应该遵循同一个闭环：

```text
目标
  -> 计划
  -> 今日行动
  -> 执行记录
  -> AI 复盘
  -> 自动调整
  -> 个人记忆沉淀
```

运动训练模块当前的闭环是：

```text
个人档案和半年目标
  -> 生成阶段计划和本周任务池
  -> 根据日程、身体状态、球类活动生成今日训练和饮食计划
  -> 用户查看动作图文教学
  -> 没有动作图时按需调用 image2 生成并保存为静态资源
  -> 用户完成训练和饮食记录
  -> 系统根据疲劳、疼痛、完成情况和球类表现调整后续计划
```

## 3. 当前模块边界

当前模块名称：

```text
运动训练与饮食一体化模块
```

它不是完整 AiLifeOS 的全部，只是第一个模块。后续可能加入学习、工作、财务、知识库、习惯、项目管理等模块，但当前不要提前实现。

当前模块应该依赖未来可复用的核心对象：

| 核心对象 | 当前模块中的作用 |
|---|---|
| Profile 个人档案 | 记录身体信息、运动背景、伤病、偏好 |
| Goal 目标 | 半年运动提升目标 |
| Plan 计划 | 阶段计划、本周任务池、今日建议 |
| Schedule 日程 | 固定球局、临时网球课、可训练时间 |
| Task 任务 | 今日训练、饮食执行、恢复任务 |
| Log 记录 | 训练、饮食、疼痛、疲劳、球类活动 |
| Asset 资源 | image2 生成的动作图、prompt、版本信息 |
| Review 复盘 | 每日、每周、阶段复盘 |
| Rule 规则 | 疲劳、疼痛、球类冲突、计划重排规则 |

## 4. 用户画像

当前优先服务的用户就是项目所有者本人：

- 健身小白，不熟悉训练动作和健身房器械。
- 刚开通半年健身卡，有明确的 24 周时间边界。
- 平时打羽毛球和网球。
- 希望提升移动、爆发、核心、肩肘稳定、下肢力量和整体运动表现。
- 需要系统告诉自己今天练什么、怎么练、为什么这么练、怎么吃、练完如何调整。
- 不需要社区、排行榜、复杂商业化功能。

## 5. 半年训练目标

半年目标不是单纯增肌或减脂，而是：

```text
通过安全、可持续的健身房训练和饮食安排，提升羽毛球和网球表现，同时建立长期训练习惯。
```

阶段建议：

| 阶段 | 周期 | 目标 |
|---|---|---|
| 适应期 | 第 1-4 周 | 学动作、熟悉器械、建立习惯、避免受伤 |
| 基础期 | 第 5-12 周 | 提升基础力量、核心稳定和关节控制 |
| 专项期 | 第 13-20 周 | 将力量、移动、旋转能力转化到羽毛球和网球 |
| 巩固期 | 第 21-24 周 | 巩固表现、降低伤病风险、总结长期策略 |

第一版优先认真做好前 4 周。后续阶段可以先保留方向，不需要一次性写死所有训练细节。

## 6. 灵活计划机制

计划不应是固定课表，而应是可重排的建议系统。

核心原则：

```text
长期目标稳定，周计划是任务池，每日计划动态生成。
```

不推荐固定写死：

```text
周一练腿，周三练胸，周五练背。
```

推荐生成本周任务池：

```text
本周需要完成：
- 下肢基础 1 次
- 上肢稳定 1 次
- 核心抗旋转 1 次
- 灵活性恢复 2 次
```

每天根据以下因素选择最合适任务：

- 今天是否有羽毛球、网球或网球课。
- 昨天练了什么。
- 睡眠和疲劳情况。
- 是否有肌肉酸痛或关节疼痛。
- 今天可训练时间。
- 本周还剩哪些任务。
- 距离本周结束还有几天。

## 7. 固定事件和临时变化

用户可能有固定球局，也可能临时约网球课，或者当天突然不想练。

这些变化必须是一等公民，而不是异常情况。

固定事件示例：

```text
每周三下班后和朋友打球。
```

临时事件示例：

```text
今天晚上约了网球课。
```

系统应该先处理固定日程和临时日程，再安排训练：

```text
固定日程
  -> 临时日程
  -> 今日身体状态
  -> 本周任务池
  -> 今日训练和饮食建议
```

用户在今日页需要能执行这些操作：

- 开始训练。
- 换一个训练。
- 改到明天。
- 今天不练。
- 我只有 20 分钟。
- 我晚上要打球。
- 身体不舒服。

每个操作都应该触发重新规划，而不是简单删除计划。

## 8. 跳过和降级机制

当用户点击“今天不练”时，系统应该轻量询问原因：

- 太累。
- 没时间。
- 身体不舒服。
- 临时有事。
- 单纯不想练。

处理规则：

| 原因 | 系统反应 |
|---|---|
| 太累 | 安排恢复，降低本周负荷 |
| 没时间 | 推荐 10-15 分钟最低可行版本 |
| 身体不舒服 | 记录疼痛或不适，避开相关动作 |
| 临时有事 | 将任务放回任务池并重新排期 |
| 单纯不想练 | 允许跳过，记录趋势，不惩罚用户 |

每个训练计划都应该有三个版本：

| 版本 | 时长 | 用途 |
|---|---|---|
| 完整版 | 45-60 分钟 | 状态好、时间充足 |
| 标准版 | 30-40 分钟 | 日常默认 |
| 最低可行版 | 10-15 分钟 | 忙、累、不想练但希望保持连续性 |

## 9. 球类运动联动

羽毛球和网球本身也是训练负荷，不能当成普通备注。

规则示例：

- 当天有高强度羽毛球，不安排重腿。
- 当天有网球课，不安排肩部高负荷训练。
- 球类后第二天疲劳高，降低健身房强度。
- 连续两天球类，第三天优先恢复。
- 肩部疼痛时减少推举、扣杀相关强化和高负荷上肢训练。
- 膝盖疼痛时减少跳跃、重深蹲、剧烈变向训练。

AI 可以解释原因，但这些安全规则应由明确规则控制，不完全交给自由生成。

## 10. 训练和饮食一体化

每日计划必须同时包含训练和饮食安排。

不是先生成训练，再给泛泛饮食建议，而是：

```text
今日训练类型、强度和时间
  -> 决定今日饮食重点
  -> 生成早餐、午餐、训练前、训练后、晚餐和饮水建议
```

今日计划结构：

```text
今日计划
  ├─ 训练安排
  │   ├─ 训练类型
  │   ├─ 动作、组数、次数、休息
  │   ├─ 新手重量建议
  │   └─ 风险提示
  │
  └─ 饮食安排
      ├─ 早餐
      ├─ 午餐
      ├─ 训练前补给
      ├─ 训练后补给
      ├─ 晚餐
      ├─ 饮水建议
      └─ 避免事项
```

饮食安排需要随训练变化同步变化：

- 如果今天不练，取消训练前后补给，改成普通日饮食。
- 如果临时增加网球课，增加训练前碳水和补水提醒。
- 如果今天是下肢训练，保证碳水和蛋白质。
- 如果今天是恢复日，保持蛋白质，碳水适中，避免暴食。

第一版不做精确卡路里和食物数据库，优先使用小白友好的结构化餐次建议。

## 11. 饮食建议原则

当前饮食目标是支持运动表现和训练恢复，不是极端减脂。

默认方向：

```text
运动表现优先，同时轻微改善体态。
```

记录方式可以使用手掌法，不要求称重：

- 蛋白质：一掌心。
- 碳水：一拳头。
- 蔬菜：一到两拳头。
- 脂肪：一拇指。
- 饮水：按杯数或大致容量记录。

第一版饮食模块不做：

- 精确热量计算。
- 大型食物数据库。
- 条形码扫描。
- 菜谱商城。
- 复杂营养素报表。
- AI 拍照识别热量。
- 严格减脂计划。

## 12. 动作教学需求

由于用户是健身小白，动作图文教学是核心体验，不是附属说明。

每个动作详情页应包含：

- 标准动作图。
- 起始姿势图。
- 发力过程图。
- 结束姿势图。
- 常见错误图。
- 训练目的。
- 对羽毛球和网球的帮助。
- 器械说明。
- 文字步骤。
- 新手重量选择。
- 常见错误。
- 停止信号。
- 替代动作。

动作说明必须用小白能理解的语言，重点回答：

- 怎么站或怎么坐。
- 手脚放哪里。
- 身体哪里发力。
- 做到什么程度。
- 什么感觉是正常。
- 什么疼痛必须停止。
- 做不了时换哪个动作。

## 13. image2 动作图生成链路

当前不预先生成全部动作图。

采用按需生成和静态资产缓存：

```text
用户点击动作
  -> 检查本地是否已有该动作图片
  -> 有：直接展示静态图片
  -> 没有：根据结构化动作数据生成 image2 prompt
  -> 本地服务或 Node 脚本调用中转站 image2
  -> 保存图片到 public/exercise-images/
  -> 更新图片 manifest
  -> 页面展示图片
  -> 后续同动作直接复用
```

图片不要存数据库。现阶段个人使用，生成图片作为项目静态资源保存，并可提交到 GitHub 仓库。

建议路径：

```text
public/
  exercise-images/
    lat-pulldown/
      cover.webp
      start.webp
      motion.webp
      end.webp
      mistake-shrug.webp
```

图片资源应有状态：

| 状态 | 含义 |
|---|---|
| missing | 没有图片 |
| generating | 生成中 |
| ready | 可使用 |
| rejected | 人工判断不准确 |
| regenerate | 需要重新生成 |

## 14. image2 安全和密钥规则

用户没有官方 image2，只持有中转站 key。中转站可能兼容 OpenAI 风格接口，也可能有差异，开发前需要先用最小脚本验证。

必须遵守：

- API key 不能写入前端代码。
- API key 不能提交到 GitHub。
- API key 放在 `.env.local` 或本地私有配置中。
- 前端不能直接调用 image2。
- 调用 image2 应通过本地 Node 脚本或本地后端服务完成。
- 生成图片后保存到项目静态目录，App 运行时只读静态图片。

需要验证的中转站信息：

- base URL。
- 模型名是否为 `image2`。
- 图片生成接口路径。
- 返回格式是 `base64` 还是 `url`。
- 支持的尺寸和参数。

## 15. Prompt 生成原则

image2 只负责画图，不负责决定动作是否正确。

动作定义必须来自结构化动作库，例如：

```text
动作名
适合等级
器械
目标肌群
动作步骤
常见错误
停止信号
替代动作
羽毛球收益
网球收益
```

再由这些字段生成 prompt。

Prompt 风格要求：

- 干净白色背景。
- 真实人体比例。
- 运动康复教学风格。
- 不要夸张肌肉。
- 不要漫画风。
- 不要复杂背景。
- 不要错误器械。
- 清晰表达起始姿势、过程或错误示范。

## 16. 页面设计方向

AiLifeOS 的界面要像个人生活操作系统，不像普通后台，也不像花哨健身 App。

视觉关键词：

```text
安静、克制、清爽、专业、有信息密度、有秩序感。
```

整体布局建议：

```text
左侧导航栏
  ├─ 今日
  ├─ 目标
  ├─ 日程
  ├─ 任务
  ├─ 记录
  ├─ 统计
  ├─ 资源
  └─ 设置

顶部状态栏
  ├─ 今天日期
  ├─ 当前目标阶段
  ├─ 今日状态
  └─ 快捷操作

主内容区
  └─ 当前页面内容
```

设计约束：

- 不做营销落地页。
- 不用大面积渐变和装饰性背景。
- 不用巨型卡片堆砌。
- 卡片圆角控制在 8px 以内。
- 少阴影，多使用细边框、分区、留白和清晰层级。
- 字体紧凑清晰，标题不要过大。
- 图标优先使用统一线性图标风格。
- 页面要适合长期高频使用。

## 17. 关键页面

第一版建议页面：

| 页面 | 作用 |
|---|---|
| 今日页 | 打开后直接看到今日训练、饮食、状态、风险和快捷操作 |
| 训练页 | 展示本周训练任务池、今日训练和可替换训练 |
| 动作详情页 | 展示图文教学、错误提示、停止信号和替代动作 |
| 饮食页 | 展示与今日训练联动的餐次安排和饮食记录 |
| 日程页 | 展示固定球局、临时课程、训练安排和冲突提醒 |
| 记录页 | 查看训练、饮食、疲劳、疼痛、球类活动时间线 |
| 统计页 | 展示完成率、训练负荷、疲劳趋势、疼痛次数、球类频率 |
| 资源页 | 管理 image2 生成图片、prompt、状态和重生成 |
| 个人档案页 | 配置身体信息、运动背景、伤病、目标和偏好 |

今日页是最重要入口。

今日页需要回答：

- 今天该不该练。
- 今天练什么。
- 今天怎么吃。
- 为什么这么安排。
- 有没有风险。
- 如果计划变了怎么调整。

## 18. 安全规则

运动训练涉及身体风险，安全层必须明确。

基础规则：

- 关节刺痛、麻木、尖锐疼痛时停止动作。
- 连续高疲劳时降低训练量或安排恢复。
- 当天有高强度球类活动时，不安排重腿。
- 肩痛时避免高负荷推举、过量扣杀相关强化和大重量上肢训练。
- 膝痛时避免跳跃、重深蹲和剧烈变向。
- 动作变形时不要加重量。
- 新手重量以“最后 2 次吃力但动作不变形”为准。

AI 可以提供解释和建议，但不能绕过安全规则。

## 19. 暂不实现范围

当前阶段明确不做：

- 用户系统。
- 云数据库。
- 社区。
- 排行榜。
- 支付。
- 多人训练计划。
- 复杂饮食系统。
- 可穿戴设备接入。
- 自动提交 GitHub。
- 前端直接调用 image2。
- 大型 AI Agent 平台。
- 全部 LifeOS 模块。

这些不是永远不做，而是当前阶段不做。

## 21. 开发优先级

推荐 MVP 顺序：

1. 产品框架和今日页。
2. 个人档案和半年目标。
3. 灵活本周任务池。
4. 今日训练和饮食一体化计划。
5. 动作详情页和结构化动作库。
6. image2 prompt 生成和本地保存链路。
7. 训练、饮食、疲劳、疼痛记录。
8. 每周复盘和下周调整建议。
9. 资源页管理生成图片和重生成状态。

## 22. 判断一个功能是否应该做

新增功能前先问：

- 它是否服务当前半年运动目标。
- 它是否能进入目标、计划、执行、记录、复盘、调整闭环。
- 它是否对健身小白有直接帮助。
- 它是否能提升训练和饮食执行率。
- 它是否会明显增加当前复杂度。

如果不能直接服务当前闭环，默认不做。
