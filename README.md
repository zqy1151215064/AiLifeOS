# AiLifeOS

AiLifeOS 是一个面向个人长期目标的 AI 生活操作系统。

它希望成为一个真正懂个人生活节奏、身体状态、目标进展和执行习惯的系统：不是简单记录事情，而是持续帮助用户把长期目标拆成今天能完成的行动，并在执行后不断复盘、调整和沉淀记忆。

## 愿景

每个人都需要一个属于自己的生活操作系统。

它应该知道你正在追求什么，理解你的时间、状态、偏好和限制，并在每天给出可执行的建议：

- 今天最重要的事情是什么。
- 今天适合推进哪些目标。
- 哪些任务应该调整、延后或降级。
- 身体状态、日程变化和长期目标之间有没有冲突。
- 做完之后应该记录什么。
- 下次计划应该如何根据真实反馈自动变化。

AiLifeOS 的目标不是替你生活，而是帮助你把生活里的长期目标持续落到每天的行动里。

## 核心理念

AiLifeOS 的底层闭环是：

```text
目标
  -> 计划
  -> 今日行动
  -> 执行记录
  -> AI 复盘
  -> 自动调整
  -> 个人记忆沉淀
```

所有模块都应该围绕这个闭环设计。

这意味着 AiLifeOS 不是功能集合，而是一个不断学习和调整的个人系统。任务、日程、训练、饮食、学习、项目和健康记录最终都应该服务同一个问题：

> 如何让一个长期目标更稳定地转化成今天的正确行动？

## V1 MVP 范围

V1 MVP 已确定为第一个完整闭环，而不是只做单点功能。

```text
训练饮食计划
  -> 灵活打卡记录
  -> 愿望激励
  -> Core 财务预算守护
  -> 手动开销记录
  -> 统计复盘
```

V1 总控交付文档见 [docs/requirements/v1-mvp-master-plan.html](docs/requirements/v1-mvp-master-plan.html)。后续进入开发前，应先阅读该文档，再进入具体模块 PRD。

技术决策：采用 monorepo、单仓库、双进程架构。V1 前端固定使用 React + TypeScript + Vite；后端服务集成在同一个仓库内，优先使用 Go。image2 代理、图片落盘、账单导入、本地 API 等服务端能力都按 Go 优先设计。

本地运行时，前端和 Go 服务分开启动：

```text
React/Vite 前端：页面、交互、IndexedDB/Dexie 本地业务数据
Go 本地服务：image2 代理、图片落盘、静态图片访问、本地 API
```

模块必须故障隔离。某个模块挂了，不能拖垮整个系统。比如 Core 财务预算检查不可用时，训练计划、打卡、动作详情和本地记录仍应可用，只是心愿兑换显示“财务模块暂不可用，稍后再确认”。

## 本地开发运行

AiLifeOS V1 采用 monorepo 组织，当前开发入口是：

```text
apps/web        React + TypeScript + Vite 前端
apps/server-go  Go 本地服务，代理 image2 并管理生成图片
packages/*      共享 schema、fixtures 和文档工具包
```

第一次运行先安装前端依赖：

```bash
npm.cmd install
```

启动 Go 本地服务：

```bash
npm.cmd run dev:server
```

启动 React/Vite 前端：

```bash
npm.cmd run dev:web
```

默认访问地址：

```text
前端：http://localhost:5173
Go 服务：http://localhost:8080
健康检查：http://localhost:8080/health
```

image2 配置只放本地环境变量或 `.env.local`，不能提交到 GitHub。可以从 `.env.example` 复制一份：

```bash
Copy-Item .env.example .env.local
```

需要配置的关键项：

```text
IMAGE_API_BASE_URL=
IMAGE_API_KEY=
IMAGE_MODEL=image2
IMAGE_OUTPUT_SIZE=1024x1024
```

如果没有配置 image2，动作详情、prompt 和本地记录仍可使用；只有生成图片接口会降级返回配置缺失。

常用检查命令：

```bash
npm.cmd run typecheck
npm.cmd run build:web
npm.cmd run test:server
```

## 完全体形态

理想中的 AiLifeOS 会包含这些核心能力：

| 能力 | 作用 |
|---|---|
| 个人档案 | 理解用户的身体状态、生活习惯、偏好、限制和长期背景 |
| 目标系统 | 管理半年、一年或更长期的个人目标 |
| 计划系统 | 将目标拆成阶段计划、周计划和今日行动 |
| 日程系统 | 处理固定安排、临时事件、时间冲突和生活变化 |
| 任务系统 | 统一管理来自不同模块的待执行事项 |
| 记录系统 | 保存训练、饮食、学习、工作、健康、情绪等真实执行数据 |
| 复盘系统 | 根据记录生成每日、每周、阶段总结 |
| 统计系统 | 呈现完成率、趋势、负荷、风险和进步 |
| 资源系统 | 管理图片、文档、Prompt、AI 生成内容和可复用素材 |
| AI 网关 | 统一管理文本模型、图像模型和后续自动化能力 |
| 规则系统 | 用明确规则约束安全、优先级、冲突处理和自动调整 |
| 个人记忆 | 沉淀用户长期偏好、反复出现的问题和有效策略 |

这些能力不应该割裂成多个孤立 App，而应该共享同一套核心对象：Profile、Goal、Plan、Schedule、Task、Log、Asset、Review、Rule 和 Memory。

## 首个落地场景：运动训练与饮食

AiLifeOS 的第一个落地场景是运动训练与饮食一体化。

这个场景来自一个真实需求：

> 用户刚开通半年健身卡，是健身小白，平时打羽毛球和网球，希望通过健身房训练、饮食安排、动作教学和复盘，让羽毛球、网球和整体运动能力持续提升。

这个模块要解决的不是“记录训练”，而是完整地回答：

- 今天该不该练。
- 今天练什么。
- 今天怎么吃。
- 为什么这么安排。
- 动作怎么做才安全。
- 如果今天不想练、没时间、临时约了网球课，计划怎么变。
- 训练和饮食执行后，后续计划如何自动调整。

## 训练计划不应该是死课表

AiLifeOS 的训练计划应该像导航，而不是命令。

现实生活里计划经常变化：可能今天突然不想练，可能每周三固定打球，可能临时约了网球课，也可能因为疲劳或疼痛需要降级。

因此理想的训练系统应该采用任务池机制：

```text
长期目标稳定
  -> 本周生成任务池
  -> 每天根据日程、疲劳、疼痛、球类活动和可用时间动态推荐
```

例如本周目标不是写死“周一练腿、周三练胸”，而是：

```text
本周需要完成：
- 下肢基础 1 次
- 上肢稳定 1 次
- 核心抗旋转 1 次
- 灵活性恢复 2 次
```

每天打开系统时，它再判断今天最适合执行哪一个任务。

## 训练与饮食一体化

如果系统生成了训练计划，就应该同步生成饮食安排。

AiLifeOS 不应该只告诉用户“今天练什么”，还应该告诉用户“今天怎么吃才能配合今天的训练”。

训练后还必须有拉伸和放松安排。对健身小白来说，训练不是做完最后一组就结束，恢复收尾会影响第二天状态、打球表现和长期坚持。

```text
今日训练类型、强度和时间
  -> 生成训练后 5-10 分钟拉伸
  -> 决定今日饮食重点
  -> 生成早餐、午餐、训练前、训练后、晚餐和饮水建议
```

例如：

- 今天有下肢力量训练，就需要保证碳水和蛋白质。
- 今天晚上有网球课，就需要训练前补充易消化碳水，并关注补水。
- 今天改成恢复日，就取消训练前后补给，保持蛋白质和正常饮食。
- 今天不练了，饮食安排也应同步调整。
- 训练完成后，系统要给出对应部位的拉伸动作、每个动作时长和停止信号。

饮食模块的长期方向不是复杂热量表，而是与目标、训练、日程和身体状态联动的日常决策系统。

## 打卡不是签到

AiLifeOS 需要一套低摩擦打卡机制，但它不应该只是“连续签到”。

打卡的目标不是制造压力，而是让系统获得足够真实的输入，用来调整今天和未来的计划。

理想中的打卡包括：

- 今日状态打卡：睡眠、疲劳、酸痛、疼痛、可用时间。
- 训练完成打卡：完成度、主观难度、疼痛、备注。
- 饮食执行打卡：蛋白质、碳水、蔬菜、饮水、训练前后补给。
- 球类活动打卡：羽毛球/网球时长、强度、表现感受、疼痛。
- 计划变化打卡：跳过、改期、缩短、替换训练的原因。
- 疼痛打卡：疼痛部位、程度、触发动作和是否停止。

跳过训练也应该是一种有效打卡。系统要知道计划为什么变化，而不是简单把它当成失败。

AiLifeOS 不应该做社交排名，也不应该用强刺激连续天数制造焦虑。更好的方式是通过打卡沉淀个人记忆，让计划越来越贴合真实生活。

## 愿望清单与盲盒激励

AiLifeOS 可以把打卡机制进一步扩展成一套跨模块激励系统。

用户维护一个真实的心愿清单，里面可以是想买的东西、想体验的活动、想去的旅行、想投入的学习资源，也可以是任何对自己有意义的奖励。

训练、学习、健康、工作等模块完成有效打卡后，系统发放卡片。卡片被收集到不同主题卡册里。当用户达到一定卡片数量或完成阶段目标时，可以开启一个盲盒。盲盒里的奖项来自用户自己的心愿清单。

这个机制的关键不是随机刺激，而是把长期坚持和真实愿望连接起来：

```text
有效打卡
  -> 获得卡片
  -> 集齐卡片
  -> 开启盲盒
  -> 命中心愿清单奖励
  -> 用户确认预算和时机
  -> 兑换或推迟
```

重要边界：

- 卡片只能通过真实行动获得，不能付费购买。
- 盲盒奖项来自用户自己的心愿清单，不做外部商业抽奖。
- 大额心愿需要预算确认，不自动触发消费。
- 如果当前预算不够，可以选择推迟、转为心愿基金进度，或在安全规则内预支未来预算。
- 恢复、降级、合理跳过也可以算有效行为，避免为了集卡而过度训练。
- 盲盒规则应透明、可控、有冷却期，不制造上瘾式循环。

## Core 财务模块：预算与预支

AiLifeOS 需要一个独立的 Core 财务模块。它不是愿望激励系统的附属页面，而是所有生活模块共享的财务状态中心。

愿望激励系统必须接入 Core 财务模块。未来学习模块买课程、健康模块买装备、旅行计划、订阅工具、阶段奖励兑换，都应该先经过同一套预算判断。

抽中心愿不等于立刻消费。系统需要先判断：

- 当前可自由支配预算够不够。
- 这个心愿是否已经有心愿基金。
- 是否会影响必要支出、固定账单和安全储备。
- 是否允许预支未来预算。
- 如果预支，未来几周或几个月如何归还。

理想流程是：

```text
抽中心愿
  -> 财务系统检查预算
  -> 预算够：可以确认兑换
  -> 预算不够：推迟 / 转为心愿基金 / 申请预支
  -> 预支通过：生成偿还计划和冷却期
  -> 预支不通过：保留心愿进度，不鼓励强行消费
```

这套机制的目的不是记账 App，而是保护用户不要因为奖励机制产生冲动消费。

Core 财务模块第一阶段应该提供：

- 财务概览：月收入、固定支出、可自由支配预算、安全储备、奖励预算。
- 预算检查：判断某个心愿、课程、装备或体验现在是否适合兑换。
- 心愿基金：把大额目标拆成可积累的资金进度。
- 预支管理：在安全规则内占用未来预算，并生成偿还计划和冷却期。
- 开销记录：第一版由用户手动添加，记录金额、分类、来源、时间、关联目标或心愿。
- 财务状态接口：给训练、学习、激励等模块返回“可兑换 / 建议推迟 / 可申请预支 / 禁止预支”等结论。

长期方向可以接入外部账单来源，例如美团、淘宝、支付宝、银行卡或信用卡账单。但第一版不做自动接入，只保留数据结构和来源字段。后续接入时必须是只读导入、用户授权、可撤销、可审计，不能自动支付或自动借贷。

## 图文并茂的动作教学

对健身小白来说，动作名没有意义，图文教学才是核心体验。

理想的动作详情页应该包含：

- 标准动作图。
- 起始姿势图。
- 发力过程图。
- 结束姿势图。
- 常见错误图。
- 器械识别。
- 动作步骤。
- 新手重量建议。
- 训练目的。
- 对羽毛球和网球的帮助。
- 常见错误。
- 停止信号。
- 替代动作。

每个动作都应该用小白能理解的语言解释：怎么站、怎么坐、手脚放哪里、哪里发力、做到什么程度、什么疼痛必须停止。

## AI 生成资源

AiLifeOS 计划使用 image2 生成训练动作教学图。

动作图片不需要预先全部生成，而应该按需生成并沉淀为可复用资源：

```text
用户点击动作
  -> 检查本地是否已有该动作图片
  -> 有：直接展示静态图片
  -> 没有：根据结构化动作数据生成 image2 prompt
  -> 调用 image2 生成图片
  -> 保存到静态资源目录
  -> 更新 manifest
  -> 后续同动作直接复用
```

重要原则：

- image2 只负责画图，不负责决定动作是否正确。
- 动作定义必须来自结构化动作库。
- 生成图片需要有可审核、可拒绝、可重生成的状态。
- API key 不能写入前端代码，也不能提交到 GitHub。

## 产品气质

AiLifeOS 的界面应该像一个个人控制台：

- 安静。
- 克制。
- 清爽。
- 专业。
- 有信息密度。
- 有秩序感。
- 适合长期高频使用。

它不应该像营销落地页，也不应该像花哨的健身 App 或数据大屏。

理想的首页不是宣传页，而是“今日入口”：今天的状态、目标、任务、训练、饮食、风险和快捷调整都应该在这里被清晰组织。

## 设计边界

AiLifeOS 不追求一开始就成为庞大的平台。

它应该先从真实个人需求出发，逐步沉淀可复用的核心能力。每个新功能都应该先回答：

- 是否服务长期目标。
- 是否能进入计划、执行、记录、复盘、调整闭环。
- 是否能减少用户决策成本。
- 是否能让系统更懂用户。
- 是否会带来不必要复杂度。

如果一个功能不能直接服务当前闭环，默认不做。

## 安全原则

涉及训练、饮食和健康的建议必须保守。

AiLifeOS 可以辅助决策，但不能替代医生、康复师、营养师或专业教练。

运动模块必须遵守明确安全规则：

- 关节刺痛、麻木、尖锐疼痛时停止动作。
- 连续高疲劳时降低训练量或安排恢复。
- 当天有高强度球类活动时，不安排重腿。
- 肩痛时避免高负荷推举和大重量上肢训练。
- 膝痛时避免跳跃、重深蹲和剧烈变向。
- 动作变形时不要加重量。
- 新手重量以“最后 2 次吃力但动作不变形”为准。

AI 可以解释规则，但不能绕过安全规则。
